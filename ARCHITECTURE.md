# Architecture

How this site works end to end — what is live vs. snapshot, where every number
comes from, how the dates update, and which pieces depend on the profile repo.
Read this before changing data flow so you don't have to re-trace it each time.

## TL;DR

- Pure static site on GitHub Pages. No build step, no framework, no backend, **no API token**.
- Two data sources at runtime:
  - **Live** — fetched in the browser from public GitHub API (max 2 requests, cached 45 min).
  - **Snapshot** — `stats.json`, the "heavy" metrics that would burn the API limit.
- `stats.json` is regenerated by a daily GitHub Action that **parses the profile's
  `terminal.svg`** — it does not compute anything itself.
- The real source of truth for all heavy metrics lives in **another repo**:
  `idesyatov/idesyatov` → `assets/terminal.svg`.

## File map

```
index.html                       # markup only; all data injected by JS at runtime
assets/style.css                 # Tokyo Night theme, responsive, a11y
assets/main.js                   # live GitHub data + cache + interactive terminal
stats.json                       # snapshot of heavy metrics (the only data file)
.github/scripts/build-stats.mjs  # rebuilds stats.json from terminal.svg
.github/workflows/refresh-stats.yml  # daily cron that runs the script and commits
.nojekyll                        # serve files as-is (no Jekyll)
```

## Data: live vs. snapshot

Unauthenticated GitHub API is capped at **60 requests/hour per IP**. Heavy
aggregates (language %, code size, commit/PR counts) are therefore **never**
computed in the browser. The split:

### Live (browser, `assets/main.js`)

Hard budget: **max 2 GitHub API requests** per load, cached in `localStorage`
(`TTL = 45 min`). Everything degrades gracefully — if a request fails or is
rate-limited, the page falls back to the snapshot/placeholder and never breaks.

| What | Endpoint | Used for |
|---|---|---|
| Profile | `GET /users/idesyatov` | `followers`, `public_repos` |
| Repos | `GET /users/idesyatov/repos?sort=pushed&per_page=100` | project cards |

Project card order: `FEATURED` list first (curated, in `main.js`), then remaining
non-fork repos that have a description, by recent push. Forks and archived repos
are dropped (`orderRepos`).

Other live calls are **on-demand only** (triggered by the mini-terminal, not on
load, and not GitHub API so they don't touch the 2-request budget):

| Command | Source | Notes |
|---|---|---|
| `whoami` | `ipwho.is` → `geojs.io` → `ipapi.co` | first reachable provider wins; 4 s timeout each |
| `news` | `hn.algolia.com` (Hacker News) | latest LLM stories, cached |
| `stars` | `GET /users/idesyatov/starred` | recently-starred repos |

### Snapshot (`stats.json`)

The only committed data file. Shape:

```jsonc
{
  "_comment": "...",
  "updated":  "2026-06-26",   // = the profile SVG's own stamp; see "Dates" below
  "languages": [ { "name": "Go", "pct": 72, "color": "#00ADD8" }, ... ],
  "code": {
    "languages":    13,    // distinct language count
    "totalCodeKB":  705,   // "Σ code" in the SVG
    "commits2026":  133,
    "pullRequests": 55,    // "PRs" in the SVG
    "commits7d":    77
  },
  "profile": { "followers": 7, "public_repos": 20 }
}
```

- `languages` + `code` — parsed from `terminal.svg` (see pipeline below).
- `profile` — **not** parsed; carried over from the previous `stats.json` and
  only used as a fallback. The live `loadProfile()` overwrites these on the page
  anyway, so they don't need to be fresh.

## The stats refresh pipeline

```
idesyatov/idesyatov : assets/terminal.svg     <-- single source of truth
            |  (raw.githubusercontent.com)
            v
.github/scripts/build-stats.mjs   (runs in CI)
            |  parse languages + code metrics, compare to old stats.json
            v
stats.json   (committed only when something changed)
            |  fetch() on page load
            v
assets/main.js : renderLanguages / renderStats / #stats-note
```

### Where each number comes from in the SVG

`build-stats.mjs` fetches the raw SVG and parses by text, **not** by a JSON feed:

- **Languages** — only the region between `$ cat languages` and `$ cat stack`
  is scanned (so neofetch text can't match). Each row = name + hex color + `NN%`.
- **Code metrics** — matched by label → the value `<tspan x="620">`:
  `languages`, `Σ code`, `commits 2026`, `PRs`, `commits 7d`.

If the SVG layout changes (labels renamed, columns moved), the parser `fail()`s
loudly and the workflow run goes red — that's the intended early-warning.

### When do the numbers actually change?

They change **only when `terminal.svg` in the profile repo is regenerated with
new values.** That regeneration is owned by the *other* repo
(`idesyatov/idesyatov`), on its own schedule. Our action just mirrors whatever
the SVG currently says. So:

- A quiet activity period → identical numbers day after day → nothing to commit.
- `commits 7d` is the most volatile field; if it's stuck, the upstream SVG
  itself is likely not being regenerated, not a bug on our side.

The SVG also carries its own stamp at the bottom, between
`<!-- UPDATED:START -->` and `<!-- UPDATED:END -->` (e.g. `updated 2026-06-26`).
That is the upstream "data was last rebuilt" date — and it is exactly what we use
as the snapshot date (see below).

## Dates — what `updated` means

`stats.json` has a single date field, **`updated`**, and it is a verbatim copy of
the **profile SVG's own stamp** (`<!-- UPDATED:START -->updated YYYY-MM-DD<!-- UPDATED:END -->`).
So the date on the page answers one honest question: *when did the upstream
pipeline last rebuild the data?*

Logic in `build-stats.mjs`:

```
parse metrics from the SVG
updated = the SVG's UPDATED stamp   (if the stamp is missing, keep the old date + warn)
write stats.json
# the workflow commits ONLY if stats.json changed:
#   - metrics differ, or
#   - the stamp moved (upstream rebuilt)
# => no empty daily commits
```

What this deliberately does **not** do: it does not stamp "today" on every run.
A cron firing is not news; an upstream rebuild is. If the profile repo doesn't
regenerate `terminal.svg`, the date stays put — which is the truth, and the
signal you want if the upstream pipeline ever stalls.

On the page (`#stats-note`, rendered in `main.js`):

```
↳ snapshot 2026-06-26 · heavy metrics are not computed in-browser (API limits)
```

> Note: the SVG stamp is **not** guaranteed to move every day — the profile repo
> rebuilds on its own schedule. So the snapshot date tracks real refreshes, not
> wall-clock days. That is intentional.

## The GitHub Action

`.github/workflows/refresh-stats.yml`:

- Trigger: `schedule` cron `17 6 * * *` (06:17 UTC daily) + manual `workflow_dispatch`.
- Top-level `permissions: {}`, job grants `contents: write` only (least privilege).
- `concurrency: refresh-stats`, `cancel-in-progress: false` — runs never overlap.
- Steps: checkout → `node build-stats.mjs` → commit & push **only if
  `git diff --quiet -- stats.json` reports a change**.
- Commits as `github-actions[bot]`.

## Invariants — do not break these

1. **No token in the repo or in `main.js`.** It would become public. Only
   unauthenticated, public requests.
2. **Max 2 GitHub API requests on page load.** Anything heavier goes through the
   snapshot or an on-demand terminal command.
3. **Graceful degradation.** Every fetch has a fallback; the page must render
   fully even with zero network.
4. **`terminal.svg` is the single source of truth** for heavy metrics. Don't
   hand-edit numbers in `stats.json` — fix them upstream and let the action mirror.
5. **The SVG parser must fail loudly** on layout changes rather than emit garbage.
